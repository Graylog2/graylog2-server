# Forms

Core rules for building forms in Graylog. This section covers container and field layout, error handling, and required field conventions.

## Layout

Forms live inside containers and are composed of fields. Both levels have rules.

<Unstyled>
  <Alert bsStyle="info">
    {
      <>
        <strong>Column Layout Rule:</strong> Forms are always laid out in a single column.
      </>
    }
  </Alert>
</Unstyled>

**Why**: CXL Institute research found single-column forms are completed 15.4 seconds faster on average. Baymard Institute usability testing found multi-column layouts cause users to skip required fields or fill unnecessary ones, drawn in the wrong direction by multiple columns of empty inputs.

### Two Columns Valid Case

A single row may use two columns when the fields are semantically one unit.

- Examples: first / last name, key / value, start / end date
- Both fields in the pair must be equal width

### Grouping and Structure

- Group related fields with a section label and divider
- Max 3 to 4 groups per form or step
- Required fields before optional within each group
- If a form has fields that most users will never need to fill out, those fields should be hidden by default behind a collapsible section labeled "Advanced options" rather than shown in the main form body

## Errors

Errors must be specific, inline, and announced to assistive technology. **Never rely on color alone.**

### Error Timing

- Validate on blur, not on every keystroke
- After a failed submit, switch to on-input for immediate recovery
- Async checks (name taken): debounce 300ms after typing stops
- Never show an error before the user has touched the field

### Error Text

- Be specific, say what is wrong and how to fix it
- Plain language, no "invalid input" or error codes
- Keep under 160 characters
- Never clear the user's input on error
